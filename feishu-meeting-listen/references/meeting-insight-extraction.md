# 从历史会议上下文提取客户洞察

## 触发条件

- 用户要求基于会议历史补充行业文档的客户洞察
- 用户说「从我旁听过的会议里提炼一下」「基于原始上下文找洞察」
- 双周会/专项文档需要补充老客信息

## 核心原则

**会议纪要 ≠ 客户洞察。** 纪要回答「发生了什么」，洞察回答「这意味着什么 + 接下来该做什么」。

差的输出：「拓竹买了 AI 包，消耗 50%」
好的输出：「拓竹 H2 目标是再卖两个 99 万 AI 包，第一个靠工具消耗、第二个靠业务场景——所有老客都会撞上这个拐点，需要组织变革咨询而非单纯增购」

## 方法论

### Step 1: 扩散（Diverge）— 从多维度铺开

从字幕中按以下 6-7 个维度系统性地提取原始证据：

| 维度 | 搜索关键词 | 要回答的问题 |
|------|-----------|------------|
| AI 组织变革 | 组织、变革、培训、agent、赋能 | 客户内部 AI 是怎么推的？谁在推动？谁在抵制？ |
| ROI/投入产出 | ROI、成本、提效、人效、降本 | 客户决策层怎么衡量 AI 的投入产出？ |
| 决策链/权力 | 老板、CEO、CTO、审批、信安 | AI 预算谁说了算？否决权在哪？ |
| 竞品/替代 | 钉钉、企微、cursor、JIRA | 客户在用什么替代方案？为什么选/不选飞书？ |
| 使用痛点 | 不够智能、不好用、不支持 | 客户对飞书 AI 的真实抱怨是什么？ |
| 增购/扩展 | 增购、升级、加包、99 | 增长的信号和障碍分别是什么？ |
| 行业对标 | 同行、友商、安克、致欧 | 客户怎么看待同行的 AI 实践？ |

**实操命令**：
```bash
# 列出所有有字幕的会议
for f in ~/meeting_logs/*.jsonl; do
  python3 -c "
import json; transcripts=[]
with open('$f') as fh:
  for l in fh:
    d=json.loads(l.strip())
    if d.get('type')=='transcript': transcripts.append(d)
print(f'{len(transcripts)}条 {f}')
"
done

# 按客户名维度提取相关字幕
python3 << 'EOF'
import json, os
CLIENTS = ["拓竹","徕芬","疆海","智能派","安克","和生"...]
for fpath in os.listdir(os.path.expanduser("~/meeting_logs")):
    ...
EOF
```

### Step 2: 收敛（Converge）— 提炼结构性洞察

每条洞察必须包含三个要素：

```
信号（从字幕中提取的具体证据）
  ↓
含义（这个信号对业务的意义）
  ↓
行动（基于此，接下来该做什么 + 如何复用到新客）
```

**质量检查**：如果一条洞察不能让读者立刻知道「接下来该干什么」，就不是合格洞察。

### Step 3: 结构化输出

每条洞察按以下格式交付：

```markdown
### 洞察 X：<一句话命名>

**证据**：<引用具体会议中的原文或场景，标注来源>

**含义**：<这个模式意味着什么，为什么值得关注>

**行动**：
- 对老客：<具体推动动作>
- 对新客：<打单时可复用的素材或话术>
```

## 常见错误

| 错误 | 正确做法 |
|------|---------|
| 输出会议纪要式的对话流 | 提取跨会议、跨客户的共性模式 |
| 只描述「他们说了什么」 | 回答「这意味着什么 + 怎么办」 |
| 洞察没有行动建议 | 每条洞察必须带 `行动` 段 |
| 忽略老客对新客的反哺价值 | 标注「此洞察对新客打单的复用方式」 |
| 问用户「帮我回忆一下」 | 自己去 `meeting_logs/` 翻，不要让用户替你回忆 |
