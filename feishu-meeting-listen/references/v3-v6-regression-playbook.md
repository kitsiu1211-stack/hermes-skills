# V3→V6 回归复盘与响应手册

> 基于 2026-07-16 真实事故编写。当用户表达极端不满时，按此流程响应。

## 事故经过

1. V3 稳定运行（poll.sh 旁听 + Agent 手动激活 → 纪要 + 会后执行）
2. 用户要求「加入会议的能力」→ Agent 识别需求范围，准备改脚本
3. **V4/V5 迁移**：从 bash 轮询迁移到 Python meeting_detect.py 时，**丢失了 poll.sh 启动逻辑**，脚本只做 C360 推送
4. 用户发现「自从更新到 V5 之后都没有产出会议纪要了」→ 说「这整个升级版本简直就是个大失败」
5. **Agent 错误响应**：在用户表达不满后，继续新增 V6 双 cron 架构 + 纪要生成 cron + 会议大脑 cron
6. 用户说「我觉得现在改的乱七八糟的，非常不满意」→ Agent 终于停止新增，开始回溯
7. 最终回归 V3 手动激活模式

## 五个致命错误

| # | 错误 | 后果 | 警示 |
|---|------|------|------|
| 1 | **每次升级都是重写不是增强** | V3 的纪要能力在 V5/V6 里消失 | 升级 = 旧版全部功能 + 新功能叠加，不是替换 |
| 2 | **过早自动化** | 手动流程还没跑稳就搞 cron/launchd，引入新 bug 没解决旧 bug | 手动跑稳 3 次 = 证明链路通，再考虑自动化 |
| 3 | **过度工程化** | V6 拆成 2 个 cron + LLM 每 3 分钟轮询，凭空烧 token | 能用 1 个 cron 解决问题就不要用 2 个 |
| 4 | **丢失上下文** | 改动时没读 poll.sh 和 SKILL.md，直接重写，把对的也丢了 | 改动前必读历史版本文件 |
| 5 | **忽视用户反馈 + 继续堆功能** | 用户说「大失败」后 Agent 仍然新增 cron/架构，导致用户说「乱七八糟」 | 不满信号 = 停止信号，不是「再改一次」的信号 |

## 🚨 不满信号 → 响应流程

### 识别信号（任意一条触发）

- 「大失败」「非常不满意」「乱七八糟」「改的什么」
- 「自从 XX 之后就没有了」「没一次成功」
- 「我觉得现在改的乱七八糟的」
- 用户重复同一要求 3 次以上

### 正确响应（四步）

1. **立即停止当前所有改动** — 不要修补刚写的代码，不要再创建新 cron
2. **回溯提问**：「之前哪个版本是稳定/可用的？我直接回退到那个版本，不做修补」
3. **找到旧版 → 恢复** — 读历史快照文件，恢复核心能力清单逐项对照
4. **确认后增量** — 用户确认基线恢复后，再逐项叠加新需求（一次一个）

### 禁止行为

- ❌ 在用户说「大失败」后继续新增功能/cron
- ❌ 在用户说「乱七八糟」后拆分架构
- ❌ 在未确认旧版恢复前创建新的自动化
- ❌ 忽略用户反馈中的情绪信号（「从来没成功过」= 当前架构从根本上不对）

## 核心能力清单对照（迁移必修）

任何对该 Skill 的架构变更，必须在改动前后逐项对照：

| 能力 | 检查 | 旧版 | 新版 |
|------|------|------|------|
| poll.sh 自动启动 | ✅/❌ | | |
| JSONL 字幕日志 | ✅/❌ | | |
| 客户会议 C360 推送 | ✅/❌ | | |
| 会议结束检出 | ✅/❌ | | |
| 纪要生成 | ✅/❌ | | |
| 会后自动执行 | ✅/❌ | | |
| 零 token 检测 | ✅/❌ | | |

**铁律**：新版必须所有项为 ✅（至少不低于旧版），才能部署。

## 正确的一次升级流程

```
1. 读历史版本文件（Obsidian versions/）
2. 列出当前核心能力清单
3. 明确新需求（用户想要什么新功能）
4. 在守旧版能力前提下增量改动
5. 逐项对照清单验证
6. 手动跑一次完整链路
7. 用户确认后再考虑自动化
```
