# ByteView 会议兼容性矩阵

> 2026-07-23 亚洲商学院分享日实测总结

## 核心发现

ByteView 实时音频 **不是所有正式会议都支持**。需要两个条件同时满足：

1. **智能体入会开关**（会议设置 → 安全 → 「允许智能体加入会议」）→ 控制 Bot 能否进会
2. **会议模板的媒体能力**（会议创建时决定）→ 控制 ByteView 是否推流

这两个开关**互不相关**。开了 1 没开 2 → Bot 能进会但听不到声音。

## 错误码速查

| 错误码 | API | 含义 | 处理 |
|--------|-----|------|------|
| 121016 | +meeting-join | 「允许智能体加入会议」开关关闭 | 让 owner 在会议安全设置中开启 |
| 121011 | realtime/endpoint | Bot 不在会议中 | 先入会再调实时端点 |
| 99991400 | +meeting-events | 频率限制 | 轮询间隔 ≥ 10s |

## 诊断流程

```
会议入会 → 启动 main.py (sleep 3)
  ├─ [bv] raw msg 出现 → 有音频 ✅
  └─ [bv] raw msg 从未出现 → 检查
      ├─ lark-cli vc +meeting-join 是否 ok？
      │   ├─ 121016 → 让用户开开关
      │   └─ ok → 继续
      ├─ realtime/endpoint 是否返回 websocket_url？
      │   ├─ 无 → 会议不支持实时媒体
      │   └─ 有 → WebSocket 层有问题
      └─ 结论：如果 Bot 入会 + WS 连接 + session created 都成功但无 raw msg
         → 会议模板不支持 ByteView 媒体流，降级字幕旁听
```

## 工作建议

- **需要语音**：统一用「袁鑫杰的视频会议」模板创建
- **正式会议无音频**：降级为字幕旁听 + 会议聊天框文字回复
- **启动时序**：`+meeting-join` → `sleep 3` → `main.py`（已验证 8 次失败→1 次成功的关键变量）
