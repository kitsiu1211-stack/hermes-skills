# 客户 AI 场景需求整理

## 架构

```
会议字幕
    │
    ▼
┌─────────────┐
│  Generator  │  第一阶段：尽可能全地提取候选场景
└──────┬──────┘
       │ 候选场景列表（带证据）
       ▼
┌─────────────┐
│  Evaluator  │  第二阶段：逐条校验、去重、合并、定优先级
└──────┬──────┘
       │ 终版场景列表
       ▼
   发送到飞书 Home
```

---

## Phase 1: Generator（场景提取）

### 目标
从字幕中不遗漏地提取所有 AI 场景。宁可多抓，不要漏。

### 步骤
1. 读取完整字幕文件
2. 逐段扫描，识别以下信号词触发的段落：
   - 场景描述类："做了"、"在用"、"打通了"、"接入了"、"搭了"、"上了"
   - 需求表达类："想做"、"需要"、"希望"、"要搞"、"能不能"、"有没有"、"计划"
   - 痛点描述类："太慢"、"限制"、"做不了"、"没办法"、"太复杂"
3. 对每个信号段落，提取候选场景：
   - 场景名称：10 字以内的一句话概括
   - 分类标签：已实现 | 需求
   - 具体描述：2-3 句，包含工具/平台/实现方式
   - 证据：引用的字幕行号范围

### 输出格式（Generator 内部）
```
候选 #1
  名称：内部服务台
  标签：已实现
  描述：AI+知识库，用户问→找不到→转人工→沉淀回知识库，含评价和提建议
  证据：283-301

候选 #2
  名称：VOC 通用打标工具
  标签：需求
  描述：用户提供数据源+规则文档→自动按规则打标，广告/人群/商品打标全用一套
  证据：403-458
```

### 注意事项
- 同一场景被多人提及 → 合并为一个候选
- 发言人为飞书侧（慕子柯）展示的案例 → 如果客户表示认可/有兴趣 → 记为需求
- 发言人为飞书侧展示但客户未回应 → 不录入
- 纯技术讨论无业务场景 → 不录入

## Phase 2: Evaluator（场景校验）

### 目标
逐条验证候选场景，去重合并，输出终版。

### 校验规则

| 检查项 | 规则 | 违规处理 |
|--------|------|----------|
| 证据校验 | 场景描述必须能在字幕中找到直接依据 | 删除 |
| 分类校验 | "已实现"必须是客户目前在生产环境使用的 | 降级为"需求" |
| 去重合并 | 同一场景的不同描述合并为一条 | 合并，合并更完整的一版 |
| 优先级 | 客户明确表达紧迫/排期的→高；提及但未排期→中；飞书展示客户仅表示兴趣→低 | 自动标注 |
| 范围校验 | 排除纯组织分析、ARPU 数据、定价讨论等非场景内容 | 删除 |
| 交叉检查 | "已实现"和"需求"间不能有同名/同质场景重复 | 保留需求侧版本 |

### 输出格式（终版）

```
【已实现 AI 场景】
1. 场景名称：xxx
   具体描述：xxx
2. ...

【AI 场景需求】
1. 场景名称：xxx
   具体描述：xxx
   优先级：高/中/低
2. ...
```

## 发送

终版输出以**纯文本**发送到用户飞书 Home 频道（`oc_e2f79ec1614a1efe1ebcd7c679bb45a8`），使用 `lark-cli im +messages-send --text`。

**不要用飞书卡片**——多维表格智能体无法读取卡片内容。

当前阶段：A2A 未实现，用户手动转发纯文本给多维表格智能体。
