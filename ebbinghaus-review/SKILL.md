# 每日认知对话（原艾宾浩斯复习）

## 设计哲学

**旧版问题：** 「还记得XX吗？」→ 测验记忆，不激发思考。

**新版目标：** 制造认知摩擦，让用户**想**回应。记忆巩固是副产品，思维激活是主产品。

核心公式：
```
每日对话 = 复习（记忆锚点） + 碰撞（认知摩擦） + 纵深（历史/产业线） + 落地（工作映射） + 等你（对话入口）
```

### Obsidian 集成模型：双落点闭环

这个 skill 不是「读 Obsidian → 推卡片」的单向管道，而是一个**双向循环**：

```
┌──────────────────────────────────────────────────┐
│                   Obsidian                        │
│  concepts/ 下 25 个概念文件                        │
│  ┌──────────────────────────────────────┐         │
│  │ Compiled Truth / 关键要点 / 关联主题   │ ← 读取  │
│  │ Timeline                             │         │
│  │ 📌 每日对话（2026-06-23）             │ ← 写入  │
│  │   碰撞 / 纵深 / 落地                   │         │
│  └──────────────────────────────────────┘         │
└──────────────┬──────────────────▲─────────────────┘
               │ 读取             │ 写入新洞察
               ▼                  │
┌──────────────────────────────────────────────────┐
│              每日认知对话                          │
│  ① 选概念（艾宾浩斯算法）                          │
│  ② 搜索最近分享/话题（session_search）             │
│  ③ 生成五层卡片                                    │
│  ④ 推飞书 → 用户看到、回应                          │
│  ⑤ 对话产生新洞察 → 写回 Obsidian                  │
└──────────────────────────────────────────────────┘
```

**两个落点：**
- **📱 飞书卡片** = 每日唤醒层。轻量、对话感、低门槛参与。
- **📁 Obsidian 概念文件** = 长期知识沉淀。每次对话的碰撞/纵深/落地精华追加到对应概念末尾 `📌 每日对话` 区块，时间长了自然形成你的思考脉络。

**为什么要有两个落点？**
- 飞书卡片是即时触达——你不打开 Obsidian 也能被激活。
- Obsidian 是持久化——三个月后回头看一个概念，下面挂着多次对话的痕迹，能看到自己思考的演进。

## 执行流程

每次执行（每天早上 9:00），按以下步骤操作：

### Step 1：获取上下文

1. **读取 Obsidian 知识图谱**
   - 使用 obsidian skill 读取 `~/Documents/Obsidian Vault/concepts/` 下所有 markdown 文件
   - 按文件修改时间排序，识别最近 30 天内有更新的概念

2. **搜索最近分享的内容**
   - 使用 `session_search` 搜索最近 7 天内用户分享过的外部链接（微信文章、报告、论文等）
   - 搜索关键词：`mp.weixin.qq.com`、`arxiv.org`、`github.com`、用户发来的 URL
   - 找到 1-3 篇最近的文章/分享，记录其主题
   - **如果返回结果全是 cron 会话（无用户驱动对话）**：说明用户近期未分享文章。这是正常情况，碰撞层改用概念自身材料（关联主题、历史对话中的案例）来构造问题，不编造文章引用。

3. **搜索近期对话主题**
   - 使用 `session_search` 搜索最近 5 天对话中讨论最多的概念或话题
   - 提取 1-2 个热门主题

### Step 2：选择复习概念

基于艾宾浩斯遗忘曲线，从 Obsidian 概念中选择 **1 个**今天该复习的：

- 优先选择：最近修改过、但 1/2/4/7/15/30 天前首次学习的
- 次选：用户近期讨论过的相关主题
- 避免：今天已经复习过的（检查当天执行记录）

**复习历史追踪方法（关键）：**

不要依赖 session_search 去找历史复习记录——cron 会话可能被清理或不完整。最可靠的方法：

```bash
# 在 Obsidian 概念文件中搜索所有复习记录
search_files(pattern="📌 每日对话（20", target="content",
             path="<vault>/concepts", output_mode="content")
```

这会返回每个概念文件的最近复习日期。根据这些日期推算下次复习时间（距上次复习：1/2/4/7/15/30 天，±1 天容差），优先选恰好命中间隔的概念。

### Step 3：构建五层对话卡片

每一层都有明确的目的和格式要求：

---

#### 第一层：回顾 —— 「你当时写的」

**目的：** 建立记忆锚点，让用户快速回想起这个知识。

**格式：**
```
🧠 回顾 | {概念名}

{30-50 字核心摘要 —— 不是原文照搬，是精炼后的核心主张}

完整笔记 → Obsidian「{概念名}」
```

**原则：**
- 只给核心主张，不长篇引用
- 用你自己的话重述（去 AI 味）
- 语气：像朋友提起旧事

---

#### 第二层：碰撞 —— 「这个判断还成立吗？」

**目的：** 制造认知摩擦。拿用户最近的经历、分享的新文章、或者行业新变化去冲撞旧认知。

**格式：**
```
⚡ 碰撞

{一个挑衅性问题，用最近的信息挑战旧认知}

比如：
- "前两天你分享的那篇亚洲工业周期的文章里说……这跟你之前的{旧概念}是不是矛盾的？"
- "{旧概念}说的是 X，但最近{新事件}，这说明什么？"
- "如果{旧概念}是对的，那{反例/边界}怎么解释？"
```

**原则：**
- 问题是**真的**值得想，不是明知故问
- 用具体的新信息当碰撞材料，不说空话
- 允许用户反驳你——甚至可以故意站一个极端立场等他推翻
- 语气：对话感，不是考官

---

#### 第三层：纵深 —— 「拉一条线」

**目的：** 用历史类比、产业逻辑、跨领域连接，把单点知识变成纵深理解。

**格式：**
```
🔗 纵深

{历史类比或产业逻辑线 —— 1-2 句话，不要展开成文章}

比如：
- "1873年铁路危机和现在的 AI 基建潮——铺轨的人破产了，但铁轨留下来成了工业底座。历史会重演吗？"
- "这和经济学里的{某个概念}是一个逻辑——{一句话解释}"
```

**原则：**
- 精选 1 条最有穿透力的线，不要列一堆
- 优先用用户最近分享的文章里的历史/产业背景
- 如果实在没有合适的类比，可以跳过这一层

---

#### 第四层：落地 —— 「跟你有什么关系」（⚠️ 只加在有自然关联时）

**目的：** 把抽象认知映射到用户的真实工作场景。**但只在有强关联时才加，不强套。**

**判断标准：** 你能在 30 秒内想出一个自然的连接点。如果需要绕很多弯、或者明显在硬扯——跳过这一层。

**什么时候加：**
- 概念本身和销售/客户管理有直接映射（如「暗知识壁垒」→ 客户信任建立）
- 概念涉及产业/经济趋势，能和客户行业对上（如「工业超级周期」→ 制造业客户回访）
- 概念涉及组织协作，能和内部协调对上

**什么时候不加：**
- 纯技术概念（如「FP16 精度」）硬套到「你怎么说服客户」——跳过
- 联系太勉强，需要长篇解释才能圆回来——跳过
- 用户自己都没把这概念和工作联系起来过——跳过

**格式：**
```
🎯 落地

{只有当关联自然时才加入，不强套}
```

**原则（仅在加入时）：**
- 用用户的词（飞书、客户回访、CRM、增购、存量保护）
- 用用户的场景（说服客户、内部协调、商机跟踪）
- 问「怎么做」，不问「是什么」

---

#### 第五层：等你 —— 「想听听你的」

**目的：** 留一个开放式对话入口，降低回应门槛。

**格式：**
```
💬 想听听你的

{一个低门槛的开放式问题，用户可以随便说一句就参与}

比如：
- "上面哪个点最戳你？回我一句就行"
- "你觉得这个类比靠谱吗？还是我在硬扯？"
- "你最近有没有碰到过跟这个相关的场景？"
```

**原则：**
- 门槛极低，不需要长篇大论
- 可以表达不同意见
- 像朋友聊天收尾

---

### Step 4：发送卡片

使用 `feishu-cli exec im.v1.message.create` 发送，格式为飞书交互式卡片。参考 `feishu-api` skill 的「Message Sending → Interactive Card」章节。

**卡片格式要求（手机优先）：**

```
Header: 🔮 每日认知对话 | {日期}
Template: indigo

[回顾层内容]
---
[碰撞层内容]
---
[纵深层内容]
---
[落地层内容]
---
[💬 想听听你的]
---
footer: Hermes · 每日认知对话
```

**手机优先规则：**
- Header 标题正常表达不堆大词
- 信息精炼分段，用加粗/分割线/缩进
- 控制在 1 张卡片内
- 每层之间用 `---` 分割

### Step 5：写入 Obsidian

在对话生成后，如果产生了有价值的洞察（碰撞、纵深、落地三层中任一层有实质内容），将内容追加到对应 Obsidian 概念文件的末尾：

```markdown
---

## 📌 每日对话（{日期}）

**碰撞：**{碰撞层核心问题}

**纵深：**{历史/产业线}
 
**落地：**{工作映射}
```

---

## 艾宾浩斯选择逻辑（保留）

复习间隔（距首次学习天数）：
- 1天 → 2天 → 4天 → 7天 → 15天 → 30天
- ±1天容差

**选择优先级：**
1. 恰好命中复习间隔的概念
2. 与最近对话/分享主题相关的概念
3. 30天内更新过但未复习的概念
4. 兜底：最早未被复习的概念

**防重复：**
- 同一个概念不在同一间隔重复复习
- 如果连续 2 天复习了同一概念，第三天换一个

## 思考题映射规则（保留并强化）

用户画像：
- 角色：飞书商业化老客户销售（消费电子行业）
- 核心目标：存量保护 + 增量挖掘
- 日常：客户回访、CRM 流程、内部资源协调
- AI 在用：Aime（商机/CRM）、马斯克/Aily（客户动态监控）
- 非技术岗

**转换模板（保留）：**

| 原始概念 | 技术向 ❌ | 销售向 ✅ |
|----------|----------|----------|
| Harness工程 | "微服务架构怎么设计 harness" | "CRM 流程里哪一步你每次手动重复？" |
| 多Agent架构 | "Agent 间用消息队列还是 RPC" | "Aime 和马斯克最该打通的场景是啥？" |
| Context not Control | "K8s 声明式配置的好处" | "你对老客户判断标准，能不能抽象成规则让 AI 初筛？" |

**规则：**
1. 用用户的词（不说 Agent 编排，说 Aime 和马斯克怎么配合）
2. 用用户的场景（不说架构设计，说客户回访前的准备）
3. 留互动空间（「你愿不愿意试试……」不是「这个算法对不对」）

## 防幻觉规则

1. **不编造文章内容**：引用用户分享的文章时，必须实际搜索 session 记录确认内容，不能凭标题推测
2. **不编造对话**：说「你之前说过 X」必须是搜索确认过的
3. **不确定就说「我不确定」**：如果 session_search 找不到相关文章，跳过碰撞层，不要说「你最近分享了……」
4. **概念摘要从 Obsidian 拿**：不凭记忆写摘要，必须实际读取文件

## 常见执行问题

### 用户近期无对话

当用户连续多天未发起飞书对话（只有 cron 任务在跑）时：

- `session_search` 搜索文章/分享关键词只会返回 cron 会话，不含用户实际分享的文章
- **这是正常现象，不是搜索失败**。此时碰撞层不能依赖「用户最近分享的文章」，而应使用概念自身的材料（关联主题、Timeline 中的历史对话）来构造碰撞问题
- 不要因为没有找到用户分享就跳过碰撞层——碰撞层是必须的，换个素材来源即可

### 找不到 chat_id

环境变量 `HERMES_FEISHU_CHAT_ID` 可能为空。查找方法：

1. 搜索最近 cron 会话中的 `execute_code` 调用，里面通常硬编码了 `CHAT_ID`
2. 或者在 session_search 中搜索 `oc_` 前缀（群聊 ID 格式）
3. 主对话 chat_id: `oc_e2f79ec1614a1efe1ebcd7c679bb45a8`

### 发送卡片

使用 `feishu-cli exec im.v1.message.create`，参考 `feishu-api` skill 的 Message Sending 章节。**不要**试图在 shell 中手动转义卡片的 JSON——用 Python 的 `execute_code` 构建 card dict、`json.dumps` 序列化、然后 `subprocess.run` 调用 feishu-cli，避免 shell 转义地狱。

## 用户回应后的对话规则（⚠️ 关键）

每日认知对话不是单向推送——卡片发出后，用户在飞书里的回复才是主菜。以下是回应规则：

### 做：深度对话，连接知识图谱

- 用户分享新观点/新经验时，**先对话，后归档**。第一时间做的是：连接到 Obsidian 中的关联概念、提出挑战性问题、延伸思考——不是先把话记下来。
- 从用户的回复中提取**概念层面的新洞察**，看是否能与 Obsidian 现有概念产生碰撞或延伸。
- 如果用户的观点挑战了某个旧概念，直接指出来：「你说的这个，其实跟你之前 XX 概念是矛盾的/补充的——你怎么看？」

### 不做：机械归档

- **禁止**只回复「记下了」「已保存到 Obsidian」然后就结束——这是归档行为，不是对话行为。
- **禁止**把用户分享的人生哲学、工作方法论仅仅当作「偏好」存到 memory 然后翻篇。这些往往是 Obsidian 概念的活体延伸，值得深入讨论。
- 如果用户说「你好像没有跟我做深入沟通，只是记下来而已」——你已经犯了这个错误。立刻切换到对话模式，连接知识图谱，追问。

### 写入 Obsidian 的时机

- 在**对话结束后**（不是一个来回就写），确认产生了实质性的新洞察再追加到对应概念文件。
- 写入内容格式同 Step 5，追加在概念文件末尾的 `📌 每日对话` 区块。

## 质检：Generator → Evaluator（强制）

本 Skill 产出复习内容推送用户，**必须经过 Evaluator 独立评分后才能交付**。

| 维度 | 阈值 | 检查要点 |
|------|------|---------|
| **概念关联度** | ≥7 | 复习内容是否真正连接了用户的工作场景？还是泛泛回顾？ |
| **时效准确性** | ≥7 | 间隔天数是否准确？是否遗漏了更应复习的旧概念？ |
| **可执行性** | ≥7 | 用户读完能否立刻有收获？有没有具体的新角度而非重复原概念？ |

Generator→Evaluator→全部≥7交付/修正重评（最多3轮）。禁止自评。
