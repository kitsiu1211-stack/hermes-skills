# Code Review

对 `HEAD` 与固定点之间的 diff 做双轴审查：

- **规范轴（Standards）** — 代码是否符合项目编码标准？
- **规格轴（Spec）** — 代码是否忠实实现了原始 issue/PRD/spec？

两轴通过 `delegate_task` 并行审查，最后汇总。

## 流程

### 1. 确定固定点

用户指定的对比点（commit SHA、分支名、tag、`main`、`HEAD~5`）。

使用三圆点 diff：`git diff <fixed-point>...HEAD`。

先确认固定点有效（`git rev-parse <fixed-point>`）且 diff 非空。

### 2. 识别 spec 来源

按优先级查找：
1. commit message 中的 issue 引用（`#123`、`Closes #45`）
2. 用户传入的路径
3. `docs/`、`specs/`、`.scratch/` 下匹配分支名/功能名的 PRD
4. 找不到就问用户

### 3. 识别标准来源

项目中的编码规范文档（`CODING_STANDARDS.md`、`CONTRIBUTING.md`）。

加上固定**代码坏味基线**（Fowler 重构 ch.3）：
- **神秘命名** — 名字不表达意图 → 重命名
- **重复代码** — 相同逻辑出现多次 → 提取
- **依恋情结** — 方法过度访问其他对象的数据 → 移动方法
- **数据泥团** — 一组字段总是一起出现 → 提取类型
- **基本类型偏执** — 用基本类型代替领域概念 → 建专属类型
- **重复 switch** — 同一类型多次 switch → 用多态
- **霰弹式修改** — 一个变更导致多处编辑 → 集中
- **发散式变化** — 一个模块因不同原因被改 → 拆分
- **夸夸其谈通用性** — 为不存在的需求加的抽象 → 删除
- **消息链** — 长链式调用 → 隐藏委托
- **中间人** — 过度委托 → 移除
- **拒绝遗赠** — 子类忽略大部分继承 → 用组合

规则：
- 项目文档覆盖坏味基线（项目标准优先）
- 始终是判断——不是硬性违规

### 4. 并行审查

使用 `delegate_task` 同时启动两个子审查：
- 规范轴子代理：检查编码标准
- 规格轴子代理：检查是否匹配 spec

### 5. 汇总报告

并列展示两轴结果。
