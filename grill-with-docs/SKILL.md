# Grill With Docs

追问式需求澄清 + 领域建模。`diverge-converge` 和 `loop-engineering-pipeline` 的共用追问内核。

## 何时触发

- 用户说了模糊需求，需要澄清
- 流水线 grill 阶段
- 用户说「grill 一下」「追问一下」

## 追问纪律

### 核心规则

1. **一次一个问题**——最关键的分叉（基础变量），它决定底下的所有设计
2. **带置信度**：如「建议方案 B，置信度 85%」
3. **对比表格**：两三个方案并列，优劣对比，最优放第一个并标出
4. **答完再进下一维**：不要十个问题一起甩

### 对比表格格式

```
| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **方案 B（推荐，置信度 85%）** | ... | ... | ... |
| 方案 A | ... | ... | ... |
| 方案 C | ... | ... | ... |
```

### 决策台账

每锁定一个决策，记录到台账：

```
| # | 决策点 | 选定方案 | 置信度 | 否决的方案 | 否决理由 |
|---|--------|---------|--------|-----------|---------|
| 1 | ... | ... | 85% | ... | ... |
```

### 追问顺序

1. **用户是谁？**（角色、技能水平、使用场景）
2. **核心问题是什么？**（痛点，不是要什么功能）
3. **成功是什么样？**（怎么知道做好了）
4. **边界在哪？**（明确不做的事情）
5. **技术约束？**（语言、平台、依赖）
6. **优先级？**（什么先做，什么后做）

## 框架摊开模式（左右脑互博）

当用户在扩散阶段说出以下触发词时，从「引导收敛」切换到「框架摊开」模式：

**触发词**：`先不收敛` `摊开所有框架` `左右脑互博` `让我看看有哪些思考角度` `框架摊开`

### 与默认模式的区别

| | 默认模式（引导收敛） | 框架摊开模式（纯扩散） |
|---|---|---|
| **输出** | 对比表格 + 推荐方案 | 框架清单，无推荐 |
| **AI 角色** | 参谋（帮你选） | 镜子（帮你看见所有视角） |
| **收敛权** | AI 建议，人确认 | 完全交还给人 |
| **适用** | 需求澄清、方案锁定 | 创意发散、突破思维盲区、审视「对得过分」的观点 |

### 输出格式

每个框架四列：

```
| 框架 | ✅ 成立条件 | ❌ 避而不谈 | ⚠️ 失效边界 |
|------|-----------|-----------|------------|
| 框架 A | 什么情况下这个框架成立 | 它故意忽略了什么 | 在什么边界下不再可信 |
| 框架 B | ... | ... | ... |
```

### 核心纪律

1. **不给推荐**——只摊框架，不帮人选
2. **至少 2-3 个互相矛盾的框架**——只有一个视角不叫互博
3. **框架之间必须有张力**——不矛盾就不值得摊开
4. **摊完后等待用户选择框架**，选定后再进入引导式收敛
5. **每个框架必须包含三列全部内容**——不能只写 ✅ 而省略 ❌ 和 ⚠️

### 何时切回默认模式

用户选定一个或多个框架后，或说出 `可以收敛了` `开始锁定` `帮我选` 时，切回对比表格 + 推荐模式。

## 领域建模

追问过程中发现领域概念时：
- 提取并标准化术语，记录到项目 `CONTEXT.md`
- 对重要决策写 ADR 到 `docs/adr/`

ADR 格式：
```markdown
# ADR-NNN: 标题

**日期**: YYYY-MM-DD
**状态**: proposed | accepted | deprecated

## 背景
## 决策
## 后果
```

## 追问结束条件

- 所有关键分叉有明确答案
- 决策台账完整（含否决的方案和理由）
- 没有遗留的「以后再说」
- 一个实现者拿到这些信息能直接开工
